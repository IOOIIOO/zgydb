## 三、新发现的问题

### 🔴 P0（阻断）— 能力评估 500，第二步完全走不通

- **现象**：`POST /ability/describe` 返回 500 Internal Server Error
- **错误**：`KeyError: 'cert_competition_label'`
- **位置**：[ability_service.py:84](file:///d:/zgyd/backend/app/services/ability_service.py#L84)
- **根因**：
  - `infer_soft_labels`（[soft_labels.py:84-94](file:///d:/zgyd/backend/app/services/real_models/soft_labels.py#L84-L94)）只返回 `logic_label` / `communication_label` / `learning_label` 三个键
  - 但 `ability_service.py` 第 82-85 行用 `labels["cert_competition_label"]` 直接取值（无 `.get()` 容错）
  - soft_labels 的 prompt 只让 LLM 推断 3 项软能力，`cert_competition` 注释为"已通过 M4 分类模型单独评估"，但 ability_service 仍从此处取值
- **影响**：
  - 能力画像无法生成，第二步完全不可用
  - 报告里 ability 部分为 None
  - P1-5（学历提取）无法验证
- **复现**：100% 稳定，每次调用必崩
- **修复方向**：ability_service 应从 `classify_certificate` / `classify_competition` 的结果汇总 `cert_competition_label`，而非从 `infer_soft_labels` 取；或将 `labels["cert_competition_label"]` 改为 `labels.get("cert_competition_label", "一般")`

### 🟡 P1 — 岗位推荐性能慢（~3 分钟）

- **位置**：[recommendation_service.py:88-110](file:///d:/zgyd/backend/app/services/recommendation_service.py#L88-L110)
- **根因**：对 top 10 岗位**串行**调用 `write_recommendation`（百炼 LLM），每次约 18s，总计约 3 分钟
- **影响**：前端极易超时（默认 fetch 超时通常 30-60s），用户等待体验差
- **修复方向**：并行调用（concurrent.futures）或减少推荐理由生成数量（如 top 5）或异步生成

### 🟡 P1 — 流程顺序约束缺失

- **现象**：能力评估（step2）未完成，但岗位推荐（step3）/ 趋势（step4）/ 报告（step5）都能正常执行
- **进度记录**：`step2_completed=False` 但 `step3_completed=True, step4_completed=True, step5_completed=True`
- **违反**：任务书明确要求"严格按顺序完成，未完成上一步不能进入下一步"
- **影响**：用户可跳过能力评估直接生成报告，导致报告数据不完整（ability 为 None）
- **修复方向**：各步骤接口入口校验上一步是否完成，未完成返回 400
