## 三、对比方案

### 当前方案：批量调用

- ✅ 减少请求次数（1 次）
- ❌ 单次请求太长（3000 字输入 + 1500 字输出）
- ❌ LLM 处理慢（30-60 秒）
- ❌ JSON 解析失败重试成本高（最坏 180 秒）

### 之前方案：并发调用

- ✅ 单次请求短（每个岗位独立，100-150 字输出）
- ✅ LLM 处理快（5-10 秒/次）
- ✅ 某个失败不影响其他
- ✅ 总耗时：max(单次) ≈ 10-15 秒
- ❌ 请求次数多（10 次）

### 推荐方案：小批量并发

- 每批 3-5 个岗位
- 并发 2-3 批
- 单次请求适中（1000 字输入 + 500 字输出）
- 预期耗时：15-30 秒

## 四、修复建议

### 方案 A：改回并发（推荐）

```python
# recommendation_service.py
with concurrent.futures.ThreadPoolExecutor(max_workers=5) as executor:
    futures = {
        executor.submit(write_recommendation, user_profile, pos_info): pos_info
        for pos_info in reason_positions
    }
    for future in concurrent.futures.as_completed(futures):
        pos_info = futures[future]
        try:
            reason_map[pos_info["position_id"]] = future.result(timeout=30)
        except Exception:
            reason_map[pos_info["position_id"]] = fallback
```

**预期效果**：10 个岗位并发，总耗时 10-15 秒。

### 方案 B：小批量 + 并发

```python
# 每批 3 个岗位，并发 4 批
batch_size = 3
batches = [positions[i:i+batch_size] for i in range(0, len(positions), batch_size)]

with concurrent.futures.ThreadPoolExecutor(max_workers=4) as executor:
    futures = {
        executor.submit(batch_write_recommendations, user_profile, batch): batch
        for batch in batches
    }
    for future in concurrent.futures.as_completed(futures):
        batch_result = future.result(timeout=30)
        reason_map.update(batch_result)
```

**预期效果**：10 个岗位分 4 批并发，总耗时 15-30 秒。

### 方案 C：优化 Embedding 批量编码

```python
# embedding.py
pos_texts = [_build_position_text(pos) for pos in positions]
pos_embs = model.encode(
    pos_texts,
    batch_size=32,
    normalize_embeddings=True,
    show_progress_bar=False
)
```

**预期效果**：100 个岗位从 8 秒降到 0.5-1 秒。

## 五、优先级

1. **最高**：修复 LLM 批量生成超时（方案 A 或 B）
2. **高**：优化 Embedding 批量编码（方案 C）
3. **中**：减少岗位数量（如只推荐 top 5）
